## 推进机制

### 单元出关
每个核心单元给两条出关线，达到任意一条即可推进：
- **标准出关：** 正常算力/状态下应达到的水平。
- **弱算力出关：** 缩小数据量、步数、模型规模后仍成立的最小验证。降级但**仍算过关**。

### 巩固关
每篇末设一道巩固关，要求你**脱稿/脱仓库**地重建或讲清本篇核心能力。**不过巩固关，不进下一篇。** 这是用"进度"替代"缓冲时间"的兜底。

### 深度档（继承 v3.1，但绑定单元而非时间）
| 档位 | 何时用 | 做到什么 |
|---|---|---|
| 保底 | 忙、累、环境/算力卡住 | 只保住「弱算力出关」+ 一个最小可验证交付 |
| 标准 | 正常状态 | 完成「标准出关」+ 该单元全部核心必做 |
| 冲刺 | 本篇核心已全过且算力充足 | 在该篇挑 ≤1 项可选附录做扩展 |

### 健康节奏
- 单个普通单元若连续 **2 个学习时段**仍没有最小可运行交付，先拆成更小的子出关，不继续扩大资料范围。
- 单个普通单元若连续 **2 周**仍未达到标准出关，优先走弱算力出关；把未解决问题写入复盘，不在原地硬耗。
- 单个巩固关允许比普通单元更久，但每周必须留下一个可检查证据：代码、流程图、复盘、录屏或答辩提纲。
- 连续 **4 周**卡在同一道关，暂停新增学习内容，只做一次外部审查：请 AI 或同伴检查卡点到底是代码能力、环境、算力、概念还是任务选型。

### 环境与算力降级
- 同一环境问题连续卡住 **2 个学习时段**仍未解决 → 记录问题，转降级路径。
- 某平台迟迟跑不通最小 demo → 不硬耗，转"调研 + 最小理解 + 替代平台"。
- **本地能转换数据格式并被训练脚本读取** 的优先级，高于 **上传 Hugging Face Hub**。

# 贯穿式 Capstone：VLA 全链路闭环（自采数据）

> **这是整份计划唯一的贯穿式项目。** 目的：亲手跑通一次 **数据收集 → 处理 → VLA 训练 → 测试** 的完整闭环，用同一份**你自己收集的数据**串起来。它**复用现有单元、不新增话题**，符合总量控制——是「加深」而非「加宽」。

**两条原则：**
- **用同一条主线：** 全程围绕篇 3（单元 7.1）选定的 LIBERO 主任务，不另起炉灶。
- **刻意做小：** 1 个带图像与语言指令的任务、一小批演示（对齐官方口径：SmolVLA 教程建议约 50 条高质量演示，其示例中 25 条已明显不足；条数打折时成功率会很低——弱算力档允许，见下方出关条件）、一个算力友好的小 VLA。**目标是「整条链路自己跑通且可解释」，不是刷高成功率。**

**四个阶段 ↔ 复用的单元：**

| 阶段 | 做什么 | 落在哪个单元 |
|---|---|---|
| ① 收集 | 在仿真里用遥操作 / 脚本专家，为选定主任务采一小批演示 | 单元 11.1 |
| ② 处理 | 把这批演示转成 LeRobot 标准格式，本地训练脚本可直接读取 | 单元 11.2 |
| ③ 训练 | 在**这批自采数据**上微调一个小 VLA（首选 SmolVLA；OpenVLA + LoRA 仅大显存备选，见单元 13.2） | 单元 13.2 |
| ④ 测试 | 闭环 rollout 评估成功率 + 失败分类，并纳入 benchmark 语境 | 单元 13.3（＋ 14.1） |

**出关条件（Capstone 专属）：**
- **标准出关：** 同一份自采数据从 ①→④ 全程打通；微调出的 VLA 能在仿真里对该任务做闭环 rollout 并报告成功率与失败分类；能讲清每个阶段的数据形态与接口。
- **弱算力出关：** 完成四件事即算过——① 数据校验（格式、形状、时间戳能被训练脚本读通）；② 单 batch 过拟合测试（loss 能压下去，证明数据流与梯度链路正确）；③ checkpoint 保存/加载可用；④ 少量闭环 rollout，且**失败分类必须完成**。**允许成功率很低甚至为 0**，但要能说清失败发生在链路的哪一环。（算力预期：官方教程 20k steps 在单张 A100 上约 4 小时，消费级显卡按数倍估；SmolVLA 对消费级硬件友好指的是「能跑」，「像样的成功率」仍需足够数据与步数——弱算力档不追求它。）
- **真机升级（可选加分 → 附录 F）：** 若拿得到 SO-100/SO-101 等低成本臂，用真机遥操作重采 ①、并在真机上重做 ④，②③ 复用。

**交付物：** 自采数据集（LeRobot 格式）＋ VLA 微调实验记录 ＋ 闭环 rollout 视频 / 成功率表 ＋ 一页《我的 VLA 数据飞轮》串讲（四阶段各自的输入输出与踩坑）。

**为什么单列：** 数据飞轮（自己产数据 → 训模型 → 评估 → 改进）是具身算法岗最稀缺、最难造假的能力证明。把它显式拎出来，避免「散落在各章、最后没真正打通」。

## 章 1｜PyTorch 项目自立 + 工程生存

**章目标：** 从空文件夹独立搭一个最小训练项目，并掌握最基础的环境、shell 与日志技能。

### 单元 1.1 环境与骨架
- **核心必做：** 配置 venv/conda、PyTorch、Git、VS Code；熟悉基本 shell（导航、管道、重定向、`tail -f`、tmux 跑后台）；建 `train.py / model.py / dataset.py / utils.py` 骨架；tensor 基础练习（shape/reshape/unsqueeze/broadcast/索引/requires_grad/backward）；实现 5 个工具函数 `set_seed / accuracy / save_checkpoint / load_checkpoint / count_parameters`。
- **标准出关：** 能解释 tensor / batch / epoch / gradient；项目骨架可运行；5 个工具函数通过自测；能逐段说明 `python x.py > out.log 2>&1 &`，会用 tmux 跑后台并 `tail -f` 看日志。
- **弱算力出关：** 同上（本单元 CPU 即可，无差别）。
- **交付物：** 项目骨架 + `tensor_playground.py` + 通过测试的 `utils.py`。
- **可选加分：** 写一个最小 Dockerfile / 环境复现脚本。

### 单元 1.2 训练最小闭环
- **核心必做：** 用 torchvision 加载 FashionMNIST，切 train/val；自己写一个小 CNN 与 forward；实现 `train_one_epoch` 与 `evaluate`；跑通若干 epoch，记录最基础的 loss / accuracy 日志。
- **标准出关：** 能说清 `zero_grad → forward → loss → backward → optimizer.step` 的顺序与各自作用（本质规则：每次 backward 前清掉上一轮梯度，且不能清在 backward 与 step 之间）；有第一版完整训练日志。
- **弱算力出关：** 数据量/epoch 数可缩小，只要闭环跑通、日志完整即算过。
- **交付物：** 第一版训练日志 + 可运行的 `train.py`。
- **可选加分：** 加 val 曲线与最佳模型记录。

### 单元 1.3 工程化 1.0
- **核心必做：** 加入 argparse 或 yaml 配置（lr / batch size / epochs / save path）；补 checkpoint 保存（latest 与 best）；输出训练曲线图；学会日志输出、stdout/stderr 重定向、基本错误定位；整理 README（环境、运行命令、目录结构、常见报错）。
- **标准出关：** 从命令行改参数即可复现实验；能独立加载 best 模型做验证；有 loss/acc 曲线图。
- **弱算力出关：** 同上（无差别）。
- **交付物：** `best.pth` + 曲线图 + README。
- **可选加分：** 实验命名规则 + 结果目录规范。

### 单元 1.4 最小实验与排错
- **核心必做：** 做 3 组小实验（不同 lr / 不同 batch size / dropout 或 weight decay 二选一）；写 1 页总结（一次 train step 到底发生了什么、为什么会过拟合）；记录最常见的 5 个 bug 及排查步骤；整理 `requirements.txt`。
- **标准出关：** 能从空目录独立重建这个项目；至少独立修掉 5 个 bug；有小实验表 + 1 页总结。
- **弱算力出关：** 实验规模可缩小。
- **交付物：** 小实验表 + 1 页总结 + bug 清单。
- **可选加分：** 最小 Dockerfile。

## 章 2｜训练脚本拆解与改写

**章目标：** 从"会写 train loop"升级到"能拆模块、能改脚本、能复现结果"。任何开源训练脚本都要能拆成数据、模型、损失、优化四层。

### 单元 2.1 实验纪律与模板
- **核心必做：** 为章 1 项目补 config、日志目录、固定随机种子、结果输出目录、实验命名规则；建实验记录模板（目标 / 改动 / 结果 / 解释 / 下一步）。
- **标准出关：** 新建任务时能快速搭起同样结构；定位问题不再只靠猜。
- **交付物：** 可复用训练模板 1.0。

### 单元 2.2 模块级理解
- **核心必做：** 把一个简单 CNN 项目拆成 data / model / train / eval / utils 五层，理清调用关系；补 scheduler、weight decay、early stopping、gradient clipping、mixed precision（先理解，不必全用）；挑一个开源训练脚本，手绘数据流与函数调用图。
- **标准出关：** 能看懂主函数入口；知道每个模块为什么存在而不是只会运行。
- **交付物：** 训练脚本地图（1 张图）。

### 单元 2.3 最小 Transformer
- **核心必做：** 用 PyTorch 自己实现 embedding、self-attention、MLP、residual、layer norm 的最小版本；做一次前向 shape 跟踪（输入 token → attention → 输出 logits）；写 1–2 页说明 attention 为何能工作、与 CNN 的差别。
- **标准出关：** 能独立解释 Q/K/V、attention score、残差连接、layer norm 的作用。
- **交付物：** `mini_transformer.py`。

### 单元 2.4 脚本改写练习
- **核心必做：** 把一个现成训练脚本缩写成最小可运行版，删掉多余框架包装；亲手改一个功能点（loss / 数据增强 / 评估指标三选一）；写复盘（删了什么、留了什么、为什么）。
- **标准出关：** 不靠 AI 逐行解释也能讲清主干流程；能独立改一个功能点并验证结果。
- **交付物：** 最小化重写脚本。

## 章 3｜微调仓库拆解与训练理解

**章目标：** 把已有的微调经验从"会跑脚本"升级到"理解数据流、参数更新和显存策略"。只做你能完整讲清楚的数据流和训练逻辑。

### 单元 3.1 脚本解剖
- **核心必做：** 选一个你已经跑过的 LoRA/PEFT 微调脚本，标出入口文件、数据整理、模型加载、可训练参数设置、训练循环；追踪一次 batch（原始样本 → tokenizer/processor → collator → forward → loss）；写注释版代码地图，只保留关键 20% 文件。
- **标准出关：** 能讲清每个 batch 的数据长什么样；知道哪些参数在更新、哪些被冻结。
- **交付物：** 微调脚本结构图。

### 单元 3.2 参数高效微调原理 + 显存
- **核心必做：** 补齐 LoRA、rank、target modules、adapter merge、梯度累积、mixed precision；做参数统计（全量 vs LoRA 可训练参数数目、显存占用预估）；学 `nvidia-smi` 与显存日志读法；看懂 lr / loss / grad norm / 显存峰值各说明什么。
- **标准出关：** 能解释为什么 LoRA 能省参数；能判断日志里哪些信号不正常。
- **交付物：** PEFT 原理笔记。

### 单元 3.3 主动改一处逻辑
- **核心必做：** 在现有微调脚本改一处关键逻辑（collator / loss / eval / save）；设计一个小对照实验确认改动是否生效；记录改动前后差异与判断依据。
- **标准出关：** 能说清"我改的不是表面参数，而是训练逻辑中的一环"。
- **交付物：** 一次可复现实验记录。

### 单元 3.4 形成方法论
- **核心必做：** 写《我如何阅读一个微调仓库》1–2 页方法笔记；整理常见故障清单（显存爆、loss 不降、数据格式错、评估不一致、保存失败）；清理篇 1 全部仓库（目录、README、requirements、运行脚本）。
- **标准出关：** 之后遇到新仓库有一套固定拆解顺序，不靠试运气。
- **交付物：** 方法笔记 + 整理版仓库。

**章目标：** 掌握三种主流 IL 范式，并统一放进同一条实验主线。核心是"做对比"，不是堆更多策略名词。

- **交付物：** Diffusion Policy 实验 + BC 对比。

### 单元 8.3 ACT (Action Chunking Transformer)
- **核心必做：** 理解 action chunking（一次预测多步动作）+ CVAE 结构；为何能减少 compounding error；训练 ACT 并与 BC、Diffusion 做三方对比；写对比表（成功率、轨迹平滑度、训练稳定性）。

- **弱算力出关：** 规模缩小，但保住三方可对比。
- **交付物：** ACT 实验 + 三方对比表。

### 单元 8.4 IL 模板收口
- **核心必做：** 整理 IL 模板（dataset / policy / train / eval / rollout / metrics）；写策略对比总结。

- **交付物：** IL 模板仓库 + 策略对比总结。

## 章 12｜感知接口（瘦身版）

**章目标：** 建立从视觉输入到"目标点 / 目标位姿"输出的**最小**前端模块，核心是"能接下游"。

> **瘦身说明：** 只做能接下游的最小感知入口；6D 位姿估计、点云、触觉下放附录 C。

### 单元 12.1 相机模型
- **核心必做：** 理解内参、外参、投影、深度图、像素 → 三维；练习：已知深度和内参把图像点恢复到相机坐标系；连接 TF/pose 直觉理解 camera frame → world frame。
- **标准出关：** 知道"检测到一个框"还不等于"知道机器人该抓哪儿"；能做像素 → 相机坐标。
- **交付物：** 相机几何 notebook。

### 单元 12.2 现成检测/分割 → 目标点位姿
- **核心必做：** 用一个现成检测或分割脚本跑通推理；把输出转成结构化对象（类别、置信度、框、mask）；定义感知前端与机器人任务之间的接口。
- **标准出关：** 能把感知结果保存成结构化对象。
- **弱算力出关：** 用轻量预训练模型 / CPU 推理。
- **交付物：** 检测/分割推理模块。

### 单元 12.3 接下游端到端
- **核心必做：** 把检测/定位结果送给下游（规划器 / 控制器 / 策略网络）；完成端到端测试：看见目标 → 生成目标点/位姿 → 下游执行；写 1 页"哪些误差最影响结果"。
- **标准出关：** 具身系统有了"感知入口"，端到端最小测试通过。
- **交付物：** 感知前端 1.0。
- **可选加分（→ 附录 C）：** 6D 位姿估计、点云、触觉感知。

## 章 13｜VLA 系统整合

**章目标：** 发挥你的 VLM/VLA 微调优势，构建一个最小的语言驱动机器人闭环。优先选算力友好的开源路线，不盲追最大模型。

> **本章总量控制：** 核心 = 选 1 条可跑 VLA 路线 + grounding + 最小闭环 demo；可选 = 更多场景变体 / 与模块化方案更深对照 / 额外架构综述。

### 单元 13.1 VLA 路线选型
- **核心必做：** 梳理 OpenVLA / SmolVLA / OpenPI（π0）等开源路线；对比 backbone、action head（autoregressive vs diffusion vs flow matching）、数据来源；理解 dual-system（快速动作 + 慢速推理）；按算力与生态选 1 条主线。
- **标准出关：** 能说清主流 VLA 的核心差异；明确为何选它。
- **弱算力出关：** 优先 SmolVLA 等小模型路线。
- **交付物：** VLA 架构对比图 + 选型理由。

### 单元 13.2 grounding / 微调 ｜ 🔗 Capstone ③·训练
- **核心必做：** **在单元 11.2 的自采数据集上微调选定的小 VLA**（首选 SmolVLA；OpenVLA + LoRA 仅在单卡显存 ≥27GB 时作为备选——官方口径缩小 batch 后仍约需 27GB，且自定义数据要接 RLDS 或自写 dataset wrapper）（= Capstone 阶段③）；把语言指令映射到目标类别 / 空间关系 / 操作约束；简单场景实现对象选择（颜色 / 类别 / 位置）；记录 grounding 失败类型。
- **标准出关：** 理解数据流与训练逻辑后主动调整，而非只跑脚本；微调在自采数据上跑通并收敛到可评估的检查点。
- **弱算力出关：** SmolVLA + LoRA/adapter + 缩小步数；数据条数可少，但按 Capstone 弱算力出关的四件套验收（数据校验 / 单 batch 过拟合 / checkpoint 存取 / 少量 rollout + 失败分类），允许成功率很低。
- **交付物：** 自采数据上的 VLA 微调实验（含可评估检查点）+ grounding 模块。

### 单元 13.3 端到端闭环 ｜ 🔗 Capstone ④·测试
- **核心必做：** 把**自采数据微调出的 VLA** 连接感知前端与下游执行；完成最小闭环 语言 → VLA → 动作执行 → 环境反馈；做闭环 rollout 评估，报告成功率与失败分类（= Capstone 阶段④）；在 2–3 个场景变体测试；记录延迟、失败点、接口不匹配。
- **标准出关：** 至少能在多个场景变体中完成任务，并给出闭环成功率与失败分类。
- **弱算力出关：** rollout 次数可少，以可解释的小样本评估为主。
- **交付物：** VLA 闭环 demo + 成功率 / 失败分类表。

### 单元 13.4 系统稳态化
- **核心必做：** 写 1–2 页架构说明（语义、感知、规划、控制各负责什么）；补异常处理与回退（目标缺失、识别错误、规划失败）；对比模块化方案（VLM 任务分解 + 经典规划执行）的优劣；整理演示视频与结果表。
- **标准出关：** 从"会微调模型"走到"会把模型接进机器人系统"。
- **交付物：** 可展示的多模态机器人 demo + 架构说明。
